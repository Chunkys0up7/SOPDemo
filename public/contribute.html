Full SOP Template

Complete structure for creating a new Standard Operating Procedure

Template Structure

All SOPs must include the following sections in this order:

  • Front Matter (YAML metadata)
  • Purpose statement
  • Scope definition
  • Compliance & regulatory requirements
  • Step-by-step procedure
  • Approval requirements
  • References & related documents
--- # Required Metadata id: sop-XX-YYY-description title: "Descriptive Title of the SOP" category: mortgage | compliance | operations | hr version: 1.0.0 effective_date: 2025-01-15 review_date: 2026-01-15 owner: Department Name approvers: - Role/Name of Approver 1 - Role/Name of Approver 2 # Compliance Mapping compliance: frameworks: - framework_id: FHA controls: [4155.1_4.A, 4155.1_4.B] - framework_id: TRID controls: [1026.19, 1026.37] # Semantic Tags tags: - underwriting - mortgage - compliance --- ## Purpose # Clear statement of why this SOP exists This SOP defines the process for... ## Scope # Who, what, when, where this applies This procedure applies to... ## Compliance & Regulatory Requirements {{include: regulatory-framework-overview}} ## Procedure ### Step 1: [Action Name] **Responsible Role:** [Role Name] **Description:** [Detailed description of the step] **Components:** {{include: component-name}} **Approval Required:** [Yes/No - if yes, specify role] ### Step 2: [Action Name] ... ## References - Related SOP: [sop-XX-YYY-name](../path/to/sop.md) - Component: [component-name](../sop-components/type/component-name.md) - External: [Regulation Name](https://url.com)

Reusable Component Template

Structure for creating modular, reusable content blocks

Component Levels

  • Atoms: Basic building blocks (definitions, single concepts, regulatory citations)
  • Molecules: Simple processes (forms, checklists, validation steps)
  • Organisms: Complex workflows (multi-step processes, approval chains)
--- # Component Metadata id: component-descriptive-name type: atom | molecule | organism title: "Component Title" description: "Brief description of what this component does" version: 1.0.0 owner: Department Name # Reusability Metadata usage_contexts: - underwriting - compliance-review dependencies: - other-component-id # Semantic Tags semantic_tags: - process - validation compliance_relevant: true frameworks: - FHA - TRID --- ## Overview # What this component does and when to use it This component provides... ## Content # The actual reusable content ### [Section Title] [Content here - can include text, lists, tables, etc.] ### [Another Section] [More content...] ## Usage Notes # Guidance for authors using this component - Include this component when... - This component should appear in [section] of SOPs - Dependencies: Requires [other-component] to be included first ## Compliance Notes # If applicable This component addresses the following regulatory requirements: - FHA 4155.1 Section 4.A: [Requirement] - TRID 1026.19: [Requirement]

Enhancement Guidelines

How to improve existing content effectively

Change Types & Version Impact

  • Patch (1.0.X): Typos, formatting, clarifications - no process changes
  • Minor (1.X.0): Additional details, new optional steps - backward compatible
  • Major (X.0.0): Process changes, new requirements - breaking changes
1

Analyze Impact

Determine if your change affects compliance requirements, approval workflows, or dependent SOPs.

2

Update Version

Increment version number according to semantic versioning based on change type.

3

Document Changes

Add clear description in your pull request explaining what changed and why.

4

Update Dependencies

If changing a component, identify and update all SOPs that include it.

Validation Before Submission

Run these checks locally before creating a pull request

# Install dependencies npm install # Run markdown linting npm run lint:md # Check for broken links npm run check:links # Validate against schema npm run validate:schema # Run all checks npm run check:all

Common Validation Issues

  • Missing required metadata fields in front matter
  • Broken internal links to components or other SOPs
  • Invalid compliance framework references
  • Incorrect semantic versioning
  • Missing approval workflow configuration
